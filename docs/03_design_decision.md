# 核心架构与设计决策 (Design Decisions)

本记录解释了数据解析引擎中关键工程代码背后的架构考量。

## 1. 为什么采用 ID 映射法 (ID Mapping)？
**决策**：不使用原始论文名称作为数据流转的标识，而是统一分配如 `P00001` 的短 ID，并维护 `paper_mapping.json`。
**考量**：
1. **系统兼容性**：彻底解决 Windows 260 字符路径限制。
2. **字符安全性**：学术论文命名常包含空格、特殊符号甚至乱码，易导致 Shell 命令注入或崩溃。
3. **下游规范化**：SFT 数据集中的 `source` 字段使用短 ID，更节省 Token 且易于溯源。

## 2. 为什么设计“目录拍平 (Directory Flattening)”机制？
**决策**：在生产模式下，脚本会潜入 MinerU 的内部目录（如 `hybrid_auto`），将 `.md` 和 `images` 提取至根目录，并删除原结构。
**考量**：
1. **防御上游黑盒**：第三方工具（MinerU）的输出结构不稳定（可能随版本、参数变动）。
2. **解耦**：下游的语义分块脚本（Semantic Chunker）不应关心 MinerU 的内部目录逻辑，只应面对最标准的数据接口（`P00001/P00001.md`）。

## 3. 为什么实现“调试/生产”双模式切换？
**决策**：在 Parser 初始化时引入 `cleanup_enabled` 开关。
**考量**：
MinerU 单次解析会产生约 20-50MB 的中间 JSON 和 Layout PDF。
* 若不清理，处理 1000 篇论文将浪费数十 GB 磁盘空间。
* 若强制清理，当模型解析出错（如公式漏框）时，开发者将失去所有 Debug 依据。
双模式设计兼顾了**可观测性（Observability）​**与**存储成本控制**。

## 4. 为什么使用 `try...except...finally` 处理单篇解析？
**决策**：将 MinerU 的 `subprocess.run` 放入严格的异常捕获与 `finally` 清理块中。
**考量**：
批处理的核心是**鲁棒性（Robustness）​**。单篇 PDF 的损坏或 OOM 绝不能导致整个流水线宕机。`finally` 确保了无论成功与否，临时工作区（`temp/`）都会被重置，防止脏数据污染下一轮循环。

## 5. 为什么放弃“固定字数切分”，改用“上下文感知切分”？
**决策**：Phase 2 采用基于 Markdown 标题的语义边界切分，并将最大长度阈值放宽至 10000 字符。
**考量**：
学术论文（尤其是无线感知领域）的长文本往往意味着完整的逻辑闭环（物理假设 $\rightarrow$ 信号模型 $\rightarrow$ 公式推导）。固定字数切分会切断推导链条和变量作用域（如上一段定义了变量，下一段才使用）。放宽长度限制并按语义切分，能够充分利用现代 LLM（如 Qwen2.5/DeepSeek）的长文本理解能力，生成具有多步推理深度的 QA。

## 6. 为什么在超长章节切分中引入“重叠滑动窗口（Overlap）”？
**决策**：当章节长度超过 10000 字符必须进行二级切分时，相邻的 Sub-chunk 之间强制保留 15% 的重叠段落。
**考量**：
如果对长章节进行“硬切断”，Part 2 会丢失 Part 1 末尾的上下文（如前置公式的结果或关键 Notation 的定义）。通过重叠滑动窗口，将 Part 1 尾部的内容复制到 Part 2 的头部，能够保证局部上下文的自洽，极大降低下游模型在生成 QA 时的幻觉率。

## 7. 为什么单独输出 `dropped_chunks.jsonl`（Debug 模式）？
**决策**：将所有被规则（黑名单、过短、过长）拦截的 Chunk 及其被丢弃的原因（`drop_reason`）写入单独的日志文件。
**考量**：
**数据清洗不能是黑盒**。我们必须知道丢弃了什么数据，才能动态调整阈值（如 `MIN_CHUNK_LENGTH`）。如果发现大量有价值的公式因为长度不足被误杀，或者有用的附录被黑名单误伤，开发者可以通过查看 `dropped_chunks.jsonl` 迅速定位问题并调整策略，这也是“工程优先于 Demo”原则的体现。

## 8. 为什么使用 Pydantic + Structured Output？
**决策**：放弃使用正则提取 Markdown 中的 ```json 代码块，转而使用 LangChain 的 `with_structured_output` 绑定 Pydantic 模型。
**考量**：
1. 彻底消灭 JSON 格式残缺问题。
2. 完美解决 LaTeX 转义灾难：传统的文本生成极易吃掉反斜杠（将 `\alpha` 变成 `alpha`），而 Structured Output 在底层 API 层面保证了转义符的完整性，保护了无线感知领域的数学公式。

## 9. 为什么引入思维链 (Chain of Reasoning) 和难度分级？
**决策**：在 Schema 中强制大模型在输出 QA 前，先输出一段内部推理（CoT），并为问题打分（1-5）。
**考量**：
对于硬核的物理原理和公式推导，如果让模型直接给出 Answer，极易产生逻辑跳跃和幻觉。强制其先“打草稿”梳理上下文，能极大提升最终 Answer 的严谨性。难度分级则为未来微调时的“课程学习（Curriculum Learning）”或数据配比提供了筛选维度。

## 10. 为什么 Schema 描述和 Prompt 全部采用纯英文？
**决策**：即使未来可能需要中文辅助，当前阶段的所有系统指令、字段描述（`description`）全部使用学术英语。
**考量**：
无线感知领域的专有名词（如 CSI, OFDM, Multipath Mitigation）在英文语境下语义最精确。全英文的约束能防止大模型在生成 JSON 时产生中英夹杂的混乱分布，确保最终获得的 SFT 数据集具备纯正的国际顶会学术英语质感。

## 11. 为什么采用“按难度降序”的贪心去重策略？
**决策**：在执行语义去重前，强制对所有 QA 按照 `difficulty` 字段进行降序排序。
**考量**：
去重算法在发现多个相似问题时，需要决定“留谁删谁”。按难度降序排序后，排在最前面的一定是包含复杂推导或深层逻辑的“终极版”问题。算法保留此项并删掉后续相似的“水题”，在去重的同时完成了一次数据质量的浓缩提纯。

## 12. 为什么在 SFT 阶段移除原始 Content？
**决策**：最终生成的 `sft_alpaca_dataset.json` 不再包含长篇原文，仅保留 `section_title` 作为 Hint。
**考量**：
为了训练“闭卷专家模型（Closed-book Expert）”。如果 Instruction 中包含原文，模型会退化为“阅读理解助手”；移除原文能强迫模型将无线感知的物理规律内化到模型权重中。

## 13. 为什么引入 `<think>` 标签？
**决策**：在 Alpaca 输出格式中，将 `chain_of_reasoning` 放在 `<think>` 标签内。
**考量**：
对齐最新的推理模型（如 DeepSeek-R1）训练范式。通过引导模型在输出答案前进行显式的逻辑建模，可以显著提升模型在处理复杂信号处理公式时的推理稳定性。

## 14. 为什么在测试集中保留 Difficulty 标签？
**决策**：在 `test_eval.json` 的 metadata 中显式保留难度等级。
**考量**：
为了实现“细粒度评估”。我们不仅要看总分，更要看模型在 Level 5（硬核推导）上的得分提升率，这比单一的准确率更能体现垂类数据的价值。
